# 프로젝트 스펙 문서 — MBTI Analyzer

## 1. 프로젝트 개요

- **한 줄 설명**: 카카오톡 대화 데이터를 분석해 특정 인물의 MBTI를 추정하는 웹 서비스
- **타겟 사용자**
  - 개발자 본인
  - 친구 및 지인
  - MBTI에 관심은 있지만 기존 검사가 번거로운 사람
- **해결하는 문제**: 기존 MBTI 검사의 번거로움 → 평소 대화로 자연스럽게 추정
- **차별점**: 대화 데이터 기반 **MBTI 특화** 분석 (레퍼런스 서비스는 일반 성격·소통 스타일 중심)
- **기간**: 2주 MVP

---

## 2. 기술 스택 & 환경

| 영역 | 선택 | 이유 |
|---|---|---|
| Frontend | **React + Vite + TypeScript** | Pydantic↔TS 타입 일치로 API 계약 버그 차단, 단일 SPA에 최적 |
| Styling | **Tailwind CSS + @tailwindcss/typography** | 마크다운 결과 렌더링이 앱 핵심 화면, `prose` 클래스로 즉시 처리. 카드 UI·반응형 최적 |
| Backend | **FastAPI (Python)** | Pydantic 모델로 API 계약 관리, Ollama Python SDK 직접 연동 |
| LLM 런타임 | **Ollama (로컬)** | 무료, 개인정보 보호, 네트워크 불필요 |
| LLM 모델 | **gemma4:e4b** | 사용자 지정 |
| 개발 환경 | macOS, Python venv, Node 18+ | - |
| 배포 | 로컬 개발 우선 | MVP 범위 외, 추후 고려 |

---

## 3. 사용자 흐름

1. 메인 페이지 접속 → 서비스 설명 + 파일 업로드 영역
2. 카카오톡 **iOS 내보내기** `.txt` 파일 업로드 → **업로드 즉시 자동 파싱**
3. 파싱된 **화자 목록** 표시 → 분석할 **화자 선택**
4. **"분석하기" 버튼 클릭** → LLM 분석 시작 (확인 의미)
5. 분석 완료 후 **마크다운 보고서** 표시 — `## 이 사람의 MBTI는 XXXX입니다` 헤더 + 성격 특징 + 축별(E/I, S/N, T/F, J/P) 근거 + 종합 해석 + 신뢰도
6. (선택) 결과 텍스트 다운로드 / 다시 분석하기

## 4. 기능별 입출력 스펙

### 4-1. 파일 업로드 & 파싱
- **트리거**: 파일 업로드 즉시 (별도 버튼 없음)
- **입력**: 카카오톡 iOS 내보내기 `.txt` (UTF-8)
- **포맷 예시**
  ```
  2023. 5. 15. 오후 2:30, 김철수 : 안녕
  2023. 5. 15. 오후 2:31, 이영희 : 안녕하세요
  ```
- **처리**: iOS 포맷 파서 → `Message[]` 변환
- **출력**: `{ messages: Message[], speakers: string[] }`
- **검증**: 샘플 파일 3개 이상에서 파싱 성공, 화자 목록 정확히 추출
- **확장성**: `ChatParser` 인터페이스로 추상화 → 향후 Android 파서 추가 용이

### 4-2. 화자 선택
- **입력**: 파싱된 화자 목록
- **출력**: 선택된 화자 이름 (state)
- **검증**: 드롭다운 선택 시 state 반영, 메시지 3개 미만이면 경고 표시

### 4-3. MBTI 분석 (핵심 기능)
- **트리거**: 화자 선택 후 **"분석하기" 버튼 클릭**
- **입력**: 선택 화자의 메시지 배열 + 타겟 화자 이름
- **처리**: FastAPI → Ollama (`gemma4:e4b`) 동기 호출
- **출력**: HTTP 200 JSON — `{ mbti, report_markdown, message_count }`
  - `report_markdown`에 다음 포함:
    - 추정 MBTI 4글자
    - 각 축(E/I, S/N, T/F, J/P) 판단 + 근거
    - 종합 해석
    - 신뢰도 코멘트
- **검증**: 메시지 20개 이상 입력 시 4글자 MBTI가 반드시 출력됨

### 4-4. 결과 표시
- **입력**: 분석 응답 JSON의 `report_markdown`
- **출력**: 마크다운 렌더링 + MBTI 카드 UI
- **검증**: 모바일·데스크탑 레이아웃 깨지지 않음, 로딩 스피너 → 완료 시 보고서 전체 표시

## 5. 의사결정 규칙

AI가 애매할 때 멈추거나 묻지 말고 *이 순서대로* 따를 기준.

### 기술 선택
- *SPEC.md에 명시된 스택 고정.* 다른 라이브러리/프레임워크로 갈아타지 말 것.
- *새 의존성 추가 전 반드시 질문.* 추가 확정 시 이유 한 줄과 함께 CLAUDE.md에 기록.
- *의존성 고민 시*: 다운로드 수 가장 많은 것.

### 코드 스타일 (공통)
- *주석은 WHY만.* WHAT은 이름으로.
- *함수 30줄 넘으면 쪼개기.*
- *에러 핸들링은 시스템 경계만* (파일 I/O, API 요청, LLM 호출). 내부 함수는 fail fast.

### 코드 스타일 (Python / Backend)
- *Pydantic 모델 사용.* FastAPI와 궁합, SPEC.md에 명시됨.
- *type hint 필수.* 가독성 용도, CI 체크는 없음.
- *로깅은 print() 또는 rich.print().* logging 모듈 안 씀.
- *try/except 중첩 금지.* 최대 1 레벨.

### 코드 스타일 (TypeScript / Frontend)
- *`any` 금지.* 불가피할 땐 `unknown` + 타입 좁히기.
- *상태 관리는 useState / useReducer만.* Redux / Zustand / Recoil 금지.
- *컴포넌트 파일 200줄 넘으면 쪼개기.*
- *Tailwind 유틸리티 우선.* 커스텀 CSS는 `prose` 오버라이드 정도만.

### 데이터 처리 (카톡 파싱)
- *파싱 실패한 줄은 스킵.* 전체 실패시키지 말 것.
- *타임스탬프 파싱 애매하면 버리기.* 분석에 필수 아님 (메시지 텍스트만 필요).
- *화자 이름 정규화 공격적으로.* 공백·특수문자 제거. 매칭 실패가 과매칭보다 나쁨.

### LLM 프롬프트
- *결과가 이상하면 프롬프트부터 수정.* 백엔드 로직 건드리지 말 것.
- *재시도/폴백 로직 금지.* 실패 시 에러 메시지 한 줄.
- *언어는 한국어 고정.* 시스템 프롬프트에 명시.

### 디자인
- *디자인 결정은 AI가 자율적으로 처리.* 사용자에게 색/레이아웃/폰트 선택을 묻지 말 것.
- *레퍼런스*: TypeMyVibe.ai, Crystal Knows. 필요시 WebFetch로 구조 참고.
- *기본 팔레트*: Tailwind `slate`(그레이 스케일) + `indigo`(포인트). 색은 최대 3가지.
- *폰트*: 시스템 기본. 웹폰트 추가 금지.
- *레이아웃*: 흰 배경 + 카드 UI 기본값. 반응형 필수 (`sm:` 이상 대응).
- *사용자 피드백 받은 후에만 수정.* 스스로 재작업 금지.

### 스코프
- *SPEC.md "하지 말 것" 목록 절대 건드리지 말 것.*
- *복잡해지면 하드코딩으로 먼저 돌아가게.* 추상화는 두 번째 구현 때.
- *`ChatParser` 인터페이스는 예외적으로 처음부터 추상화* (향후 Android 추가 대비, SPEC.md에 명시됨).

### 파일 / 폴더
- *프론트는 `src/components`, `src/lib`, `src/types` 3개 폴더만.* 서브폴더 금지.
- *백엔드는 기능별 모듈 분리.* `main.py`(라우트), `models.py`, `parsing/`(파서), `analysis/`(LLM·프롬프트).
- *파일 삭제 전 반드시 질문.*

### 판단 자체가 안 설 때
- *가장 단순한 쪽.* MVP의 적은 복잡도가 최우선.
- *그래도 모르면 질문.* 단, 질문은 명확한 선택지 2~3개로 제시할 것.
