## 2. LLM이 최근 6000자만 본다

### 현재
- `backend/analysis/llm.py`의 `MAX_PROMPT_CHARS = 6000`
- 타겟 화자 메시지를 이은 결과가 6000자 넘으면 **뒤에서 6000자만** 잘라서 사용
- 즉 LLM은 가장 최근 대화 일부만 본다
- Ollama `num_ctx = 8192` (토큰 단위)

### 문제
- 메시지가 많은 화자는 **초반·중반 대화가 전부 잘림**
- 최근 편향(recency bias): 평소 성향보다 최근 말투 위주로 판단
- 장기적인 말투 변화·패턴을 놓침
- 대략 한 화자당 메시지 400~500개부터 잘리기 시작

### 결정해야 할 것
- `MAX_PROMPT_CHARS`를 올릴 것인가? (예: 6000 → 15000)
- `num_ctx`도 같이 올릴 것인가? (예: 8192 → 16384)
- 잘라낼 때 "최근 우선" 말고 다른 방식을 쓸 것인가? (랜덤 샘플 / 앞뒤 균등 샘플 등)

## 3. 타겟 화자 메시지만 본다 (맥락 부재)

### 현재
- `analyze_mbti()`에서 `messages`를 타겟 화자로 필터링한 뒤 그것만 LLM에 전달
- 다른 사람의 질문·반응·대화 흐름은 **전부 버려짐**
- LLM은 "이 사람이 혼자 말한 목록"만 본다

### 문제
- 단답 판정 부정확 — `"ㅇㅇ"`, `"아 넵넵"`이 성실한 동의인지 귀찮은 회피인지 구분 불가
- 반응 패턴 손실 — 상대가 슬픈 얘기 했을 때 어떻게 반응하는지 안 보임 (T/F 판단 치명적)
- 대화 주도권 측정 불가 — 화제 전환을 주도하는지 따라가는지 안 보임 (E/I 중요 신호)
- **이게 MVP 결과 품질의 가장 큰 병목일 가능성이 높음**

### 결정해야 할 것
- 맥락을 얼마나 포함할 것인가?
  - (A) 전체 대화 + 화자 태그 표시
  - (B) 타겟 메시지 + 직전 N개 윈도우
  - (C) 질문-응답 쌍으로 재구성
- 프롬프트에서 타겟과 맥락을 어떻게 구분해 LLM에 알릴 것인가
- 맥락을 늘리면 위 "항목 2"의 6000자 제한과 충돌 — 같이 결정 필요
- 개인정보 관점 재검토 — 다른 사람 메시지도 LLM에 들어가게 됨
